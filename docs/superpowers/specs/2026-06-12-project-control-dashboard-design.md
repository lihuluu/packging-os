# 项目总控台可视化仪表盘设计规格（已退役 / 2026-06-15）

> 该方案已于 2026-06-15 退役。`project-dashboard.html` 和生成器 `build-project-visual-dashboard.py` 已从系统中移除。本规格保留作为历史决策记录。

## 背景

Packging OS 此前已有 Markdown 版项目驾驶舱（`project-dashboard.md`，已退役），能汇总项目、最近更新、最近产出、当前阶段、阻塞项和下一步动作，但日常使用仍需要阅读多个 Markdown 文件才能判断”今天先推进哪个项目”。

用户确认第一版仪表盘选择“项目总控台”方向，重点不是替代 Markdown，而是把已有项目事实、执行追踪和文件状态可视化，形成一个只读的本地工作台。

## 目标

做一个只读静态 HTML 项目总控台，打开后能直接回答：

- 今天优先推进哪个项目。
- 每个项目为什么排在当前位置。
- 当前最大瓶颈是什么。
- 下一步动作是什么。
- 项目卡和执行追踪表是否可信、是否落后于最近产出。

## 非目标

- 不做编辑器，不在仪表盘内修改 Markdown。
- 不引入数据库或账号系统。
- 不做长期运行的服务端。
- 不替代 `project-memory-card.md` 或 `project-tracker.md`。
- 不在第一版做复杂权限、多人协作或实时通知。

## 数据源

### 1. 项目事实源

固定读取：

```text
Workspace/Projects/{project}/00_Project_Control/project-memory-card.md
```

用于提取：

- 项目名称
- 产品/品类
- 当前阶段
- 开案日期
- 目标上市日期或硬截止日期
- 商业硬约束
- 当前最大商业风险
- 风险
- 依赖项
- 下一步动作

### 2. 执行追踪源

读取项目目录内唯一的：

```text
project-tracker.md
```

它可以位于项目内任意目录，例如：

```text
03_Design/01_Working/project-tracker.md
```

每个项目最多允许一个 `project-tracker.md`。如果找到多个，仪表盘不猜测哪一个有效，直接把该项目标记为“追踪表冲突”，并提示整理。

`project-tracker.md` 用于提取：

- 当前阶段
- 倒计时
- 关键里程碑
- 任务清单
- 关键路径
- 合规/授权/验收节点
- 供应商沟通记录
- 风险节点
- 下一周动作
- 当前最大瓶颈
- 最近更新

### 3. 文件状态源

扫描项目目录的最近产出文件，用于判断：

- 最近产出路径
- 最近产出更新时间
- `project-memory-card.md` 是否落后于最近产出
- `project-tracker.md` 是否落后于最近产出

扫描时继续排除：

- `00_Project_Control` 内自动生成的驾驶舱文件
- 临时文件
- 系统文件
- `.gitkeep`、`.gitignore`、`.DS_Store`、`Thumbs.db`、`desktop.ini`

## 数据职责边界

| 文件 | 职责 |
| --- | --- |
| `project-memory-card.md` | 项目事实、商业约束、冻结决策、风险摘要和事实级下一步。 |
| `project-tracker.md` | 推进过程、任务、里程碑、关键路径、供应商沟通和当前瓶颈。 |
| 静态 HTML 仪表盘 | 只读聚合、排序、解释和导航，不写回项目事实。 |

仪表盘的“下一步”和“当前瓶颈”优先来自 `project-tracker.md`。当 tracker 缺失时，再使用项目卡中的下一步动作和风险摘要兜底。

## 优先级评分

每个项目生成一个综合推进分数，显示为“等级 + 分数”，例如：

```text
高 82
中 56
低 24
```

分数用于排序，不代表精密项目管理算法。界面必须同时显示可解释原因。

### 建议评分维度

| 维度 | 示例信号 |
| --- | --- |
| 时间压力 | 距目标上市或硬截止日期越近，分数越高。 |
| 风险和阻塞 | 项目卡风险、tracker 风险节点、阻塞项越多，分数越高。 |
| 任务紧急度 | tracker 中存在“最高”“高”优先级未完成任务。 |
| 当前瓶颈 | tracker 中存在明确“当前最大瓶颈”。 |
| 状态可信度 | 项目卡或 tracker 落后于最近产出时加分并提示。 |
| 下一步明确度 | 有明确下一步时仍可推进，但不因模糊而降低风险提示。 |

### 排序原因示例

```text
高优先级：距上市 18 天，存在最高优先级未完成任务，当前瓶颈为外箱实地试装，tracker 早于最近产出。
```

## 页面设计

### 1. 总览页

第一屏按综合推进优先级排序。每个项目显示为一张紧凑卡片。

卡片字段：

- 优先级等级和分数
- 排序原因
- 项目名称
- 产品/品类
- 当前阶段
- 当前最大瓶颈
- 最应该做的下一步
- 状态可信度
- 最近产出
- 最近更新时间
- 快捷入口：项目卡、tracker、项目目录、最近产出

### 2. 轻量详情展开区

点击项目卡后在当前页面展开详情，不跳转到新页面。

展开区显示：

- tracker 的关键里程碑摘要
- 未完成的高优先级任务
- 关键路径摘要
- 最近一条供应商沟通记录
- 风险节点摘要
- 最近更新时间
- 数据异常提示

### 3. 异常状态

| 状态 | 界面表现 |
| --- | --- |
| 缺项目卡 | 标记为“缺项目事实源”，不参与完整评分。 |
| 缺 tracker | 显示“暂无执行追踪表”，使用项目卡兜底。 |
| 多个 tracker | 标记为“追踪表冲突”，提示整理，不自动读取。 |
| 项目卡落后 | 显示状态可信度提醒。 |
| tracker 落后 | 显示执行追踪可能过期提醒。 |
| 无目标日期 | 时间压力维度降权，提示缺少截止信息。 |

## 生成方式

第一版采用静态生成：

```text
读取 Markdown -> 解析项目数据 -> 计算优先级 -> 生成 HTML
```

推荐新增脚本位置：

```text
.claude/skills/project-memory-manager/scripts/build-project-visual-dashboard.*
```

推荐输出位置：

```text
Workspace/Projects/project-dashboard.html
```

后续可在现有 `build-project-dashboard` 流程中增加 HTML 生成，但第一版可以先独立脚本，降低对现有 Markdown 驾驶舱的影响。

## 解析策略

第一版优先采用结构化 Markdown 解析和稳定标题匹配。

必须识别的标题：

- `## 项目摘要`
- `## 商业硬约束`
- `## 风险`
- `## 依赖项`
- `## 下一步动作`
- `## 当前阶段`
- `## 关键里程碑`
- `## 任务清单`
- `## 关键路径`
- `## 供应商沟通记录`
- `## 风险节点`
- `## 当前最大瓶颈`

对无法解析的字段，不报错中断；在项目卡片中显示“字段缺失”或“无法解析”，并降低该维度的确定性。

## 验证方式

- 在当前 4 个项目目录上生成一次 HTML。
- 确认 HT2385 能读取现有 `03_Design/01_Working/project-tracker.md`。
- 确认每个项目最多一个 tracker 的规则生效。
- 确认缺 tracker 的项目仍能从项目卡生成卡片。
- 确认优先级排序显示等级、分数和原因。
- 确认项目卡、tracker、项目目录和最近产出链接可点击。
- 确认生成过程不修改任何项目 Markdown。

## 后续扩展
