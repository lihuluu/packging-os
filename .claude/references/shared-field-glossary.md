# Packging OS 共享字段词表

## 目标

统一项目卡、项目复盘、知识沉淀三类文档的公共字段，减少同义字段漂移，方便 Claude Code 在跨 skill 更新时保持口径一致。

原则：
- 同一语义尽量使用同一字段名
- 专属字段可以保留，但应建立在共享字段之上
- 不为风格变化发明新标题

## 核心共享字段

### `项目摘要`

用于快速说明“这是什么项目/本次文档覆盖什么范围”。

建议包含：
- 项目名称
- 产品/品类
- 当前阶段或复盘阶段
- 时间范围或关键节点

不要再混用：
- `项目概况`
- `项目摘要（仅复盘使用）`

### `来源文档`

用于说明结论来自哪些文件或记录。

建议包含：
- 项目总卡
- 关键阶段文档
- 复盘文档
- 关键沟通记录

不要再混用：
- `扫描文档`
- `证据来源`

### `已知事实`

用于记录已确认、可直接作为判断基础的信息。

可以是：
- 已确认事项
- 已发生事实
- 已观察到的问题

规则：
- 已确认才能写在这里
- 不确定内容移到 `关键假设` 或 `开放问题`

### `关键假设`

用于记录尚未完全验证、但足以影响判断的推断。

适用场景：
- 项目卡里的前置假设
- 复盘里证据不足的根因判断
- 知识沉淀里尚未固化的经验

不要再拆成多个近义标题：
- `待验证假设`
- `低置信假设`

如果需要表达证据强弱，在内容里注明置信度。

### `风险`

用于记录会影响成本、排期、量产、合规、协作或判断正确性的风险。

规则：
- 风险写成“风险点 + 影响”
- 不把纯待办写成风险

### `开放问题`

用于记录仍未关闭的问题、冲突信息或待确认项。

覆盖这些旧叫法：
- `开放问题`
- `冲突 / 待验证`
- `待处理项目`

规则：
- 如果是信息冲突，写出冲突来源
- 如果是待确认项，写出谁来确认

### `结论沉淀`

用于记录这份文档要保留下来的结论、规则、经验或回写内容。

覆盖这些旧叫法：
- `需要沉淀的知识`
- `新增知识`
- `更新知识`
- `建议回写位置`

规则：
- 先写“沉淀什么”
- 再写“写到哪里”

### `下一步动作`

用于记录文档完成后，下一步要做什么。

规则：
- 写成动作，不写成抽象建议
- 尽量带责任对象或目标文件

## 三类文档的最小字段集合

### 1. 项目卡

必须包含：
- `项目摘要`
- `已知事实`
- `关键假设`
- `风险`
- `开放问题`
- `下一步动作`

可保留专属字段：
- `冻结项`
- `依赖项`
- `本次更新`

### 2. 项目复盘

必须包含：
- `项目摘要`
- `来源文档`
- `已知事实`
- `关键假设`
- `风险`
- `开放问题`
- `结论沉淀`
- `下一步动作`

可保留专属字段：
- `根因判断`
- `应该保留的做法`
- `应该调整的地方`

### 3. 知识沉淀

必须包含：
- `项目摘要`
- `来源文档`
- `已知事实`
- `关键假设`
- `风险`
- `开放问题`
- `结论沉淀`
- `下一步动作`

可保留专属字段：
- `工作区级结论`
- `覆盖状态`

## 禁止的同义漂移

在没有明确必要时，不要来回切换这些标题：

| 不建议继续混用 | 统一改用 |
| --- | --- |
| 项目概况 / 项目摘要 | `项目摘要` |
| 扫描文档 / 证据来源 / 来源文档 | `来源文档` |
| 待验证假设 / 低置信假设 | `关键假设` |
| 冲突 / 待验证 / 待确认项 | `开放问题` |
| 需要沉淀的知识 / 新增知识 / 更新知识 / 建议回写位置 | `结论沉淀` |

## 使用方式

- 修改项目卡、复盘或知识沉淀相关 skill 时，优先回看这份词表
- 修改模板时，先统一标题，再调整内容结构
- 如果确实要新增字段，优先判断它是不是已有共享字段的变体
