# Packaging OS 品类策略

## 定位

Packaging OS 的底层不是单一品类工具，而是面向独立包装设计工作室的 Claude Code 工作流系统。

当前默认业务域是茶叶包装设计。茶叶不是唯一未来方向，而是第一套要跑成熟的垂类样板。后续扩展咖啡、酒类、食品、保健品或其他品类时，应新增对应品类模块，而不是推翻 9 层工作流骨架。

## 工作流骨架

这些能力保持通用：
- brief 拆解
- 竞品与定位研究
- 概念生成
- 结构选择
- 视觉系统
- 材质工艺
- 供应商沟通
- 报价比较
- 项目推进
- 提案组织
- 印前检查
- 版本追踪
- 项目复盘与知识沉淀

通用骨架只回答“包装项目如何推进”。品类模块回答“这个品类的真实约束是什么”。

## 当前默认垂类：茶叶包装

当用户没有特别说明品类时，默认按茶叶包装项目理解。

茶叶项目优先关注：
- 茶叶 SKU：货号、产品名称、茶类/工艺、规格/净含量
- 包装形态：礼盒、铁罐、纸罐、铝箔袋、卡盒、内托、手提袋、外箱、折页
- 使用场景：日常自饮、商务礼赠、节庆礼赠、电商销售、线下专卖
- 档期：中秋、春节、春茶、日常补货、电商活动
- 标签信息：产品名称、茶类/工艺、净含量、执行标准、SC、生产商、产地、贮存条件、条码等
- 生产风险：防潮、防串味、内袋食品接触、铁罐尺寸、礼盒装配、MOQ、打样周期、印前合规
- 视觉风险：茶叶包装同质化、传统符号堆叠、礼赠俗套、系列 SKU 区分不足

涉及法规、标签或食品接触安全时，只做设计与流程风险提示，不替代专业法规审核。

## 非茶叶品类处理

如果用户明确提出非茶叶品类：
- 不把茶叶字段强行套用到该品类。
- 仍可使用通用工作流推进 brief、结构、视觉、供应商和印前问题。
- 输出中标注“当前缺少该品类专用规则”，并列出必须补充的品类事实。
- 如果同类项目会反复出现，建议新增独立品类模块。

## 新增品类模块的最小内容

每个新增品类至少补齐：
- 品类 brief 字段
- 常见包装形态
- 强制信息与合规风险
- 结构和材料约束
- 竞品常态与视觉俗套
- 供应商询价口径
- 印前检查重点
- 3 到 5 条回归测试样例

## 治理原则

- 不为了当前项目把通用工作流写死成单一品类。
- 不在通用 skill 里堆大量品类细节；品类细节应沉淀到 references 或未来独立品类模块。
- 茶叶默认值可以出现在模板、示例和判断清单里，但要保留未来扩展入口。
