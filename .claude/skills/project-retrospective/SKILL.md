# Packaging Retrospective

## 概述

把包装项目发生过的事情转成以后还能用的经验。重点是找根因、保留有效做法、修正流程，而不是写流水账。

## 典型触发句

- 帮我复盘一下这个包装项目。
- 这次包装上市后反馈一般，帮我总结问题和原因。
- 请把这次项目的经验整理成 SOP 更新项。
- 我想知道这次包装项目哪些做法应该保留，哪些应该改。
- 帮我把这次打样和量产的问题沉淀成复盘文档。
- 总结一下这次包装项目哪里做得好，哪里做得差。
- 帮我把这次踩坑沉淀成以后能复用的规则。

## 使用流程

1. 收集市场反馈、生产问题和协作过程。
2. 使用 [retrospective-framework.md](./references/retrospective-framework.md) 区分现象和根因。
3. 把经验归类为：应该保留、应该改变、应该固化。
4. 写出具体的 SOP 更新项和未来检查点。
5. 结尾指出哪些知识值得继续交给 `knowledge-synthesizer` 回写到共享知识库。

## 复盘规则

- 区分单项目事故和系统性弱点。
- 每个经验都要关联到流程中的某个阶段，看本来应在哪里被发现。
- 尽量输出可执行的流程修改，而不是空泛建议。
- 不只记录失败，也要记录成功经验。
- 如果证据不足，不要硬推断根因。

## 输出要求

- 输出 `项目摘要`、`来源文档`、`已知事实`、`关键假设`、`风险`、`开放问题`、`根因判断`、`应该保留的做法`、`应该调整的地方`、`结论沉淀` 和 `下一步动作`。
- 关键问题要映射到原本应该被发现的阶段。
- SOP 更新项要直接落实到问题、规则、模板或审批点。

## 输出后检查

- 如果本次复盘新增了系统性风险、确认了冻结项代价、更新了供应商/MOQ 认知，或改变了后续项目的时间线/预算/合规判断，必须提示回写相关项目卡。
- 提示格式使用：`⚠️ 本次任务改变了 [具体项]，建议更新项目卡：project-memory-manager`
