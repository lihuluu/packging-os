# Project Tracker

## 概述

把包装项目从“很多待办”整理成可推进的里程碑和责任清单。重点是识别瓶颈、明确责任，并把依赖关系写清楚。

## 典型触发句

- 帮我排一下这个包装项目的时间线。
- 请整理这个项目接下来要做什么。
- 这个包装项目现在卡在哪个节点？
- 帮我列一个推进清单和负责人。
- 这个项目怎么安排打样、评审和发厂节点？
- 帮我把这个项目拆成里程碑和责任人。
- 现在最该推进哪一步？

## 使用流程

1. 确认当前阶段、目标上线时间、主要参与方、合规/授权/验收负责人和关键依赖。
2. 把任务拆成里程碑、子任务、审批节点和外部依赖。
3. 区分 `必须先完成`、`可并行` 和 `高风险延误项`。
4. 输出阶段计划、责任人建议和近期动作。
5. 标出最可能导致延期或返工的节点。

## 规划规则

- 先列关键路径，再补次级任务。
- 需要外部供应商、打样或审批的节点，默认都要预留缓冲。
- 不要把“等待确认”写成已完成；待批状态要单独标明。
- 如果项目跨多个部门，责任和输入输出必须写清。
- 时间线不确定时，优先给相对顺序和关键依赖，不硬写伪精确日期。

## 输出要求

- 输出 `当前阶段`、`关键里程碑`、`任务清单`、`责任建议`、`合规/授权/验收节点`、`风险节点` 和 `下一周动作`。
- 如果适合，用表格展示时间线或责任分工。
- 里程碑时间线应与项目卡保持一致，推进表中如涉及里程碑节点变更，必须同步提示更新项目卡。
- 结尾指出当前项目最需要推进的一个瓶颈。

## 输出后检查

- 如果本次任务改写了里程碑、责任分工、预算、关键依赖、供应商节点、合规/授权/验收节点或上线时间，必须提示更新项目卡。
- 提示格式使用：`⚠️ 本次任务改变了 [具体项]，建议更新项目卡：project-memory-manager`

## 默认归档位置

- 默认目录：`00_Project_Control`
- 推荐文件名：`project-tracker.md`
- 归档规则：属于持续更新的项目推进文档，应覆盖更新，不进入 `04_Final`。与项目事实卡放在同一控制层。

- project-tracker 从开案阶段就可使用，不限于后期推进
- 推进表必须包含：关键路径节点、外部等待项、审批节点、合规/授权/验收节点、下一周商业动作
