# 包装知识沉淀框架

## 目标

把项目产出提炼成未来可复用的最小知识单元，而不是重复存档项目摘要。

每条知识都应能回答这四件事：
- 这条结论在哪类场景下成立？
- 它解决什么判断问题？
- 应采取什么动作？
- 证据来自哪些项目或文档？

## 共享字段词表

知识沉淀与项目卡、项目复盘统一使用以下公共字段：
- `项目摘要`
- `来源文档`
- `已知事实`
- `关键假设`
- `风险`
- `开放问题`
- `结论沉淀`
- `下一步动作`

如需完整定义，阅读 [../../../references/shared-field-glossary.md](../../../references/shared-field-glossary.md)。

## 展示原则

知识库默认分成两层：

- 人读层：`Workspace/Knowledge/Library/knowledge-dashboard.md`、`Workspace/Knowledge/Library/workspace-knowledge-digest.md` 和各分类知识页。
- 运行层：`Workspace/Knowledge/Operations/` 下的覆盖表、收件箱、会话包和日志。

人读层优先满足这三个要求：

- 一眼能扫到结论，不先阅读一大段过程描述。
- 同类信息用表格收纳，减少重复字段堆叠。
- 先给摘要，再给分类明细。

## 知识分类

### 1. 决策规则

适合写成“如果...那么...”的判断规则。

示例：
- 如果结构未锁定，不进入视觉定稿。
- 如果供应商能力未确认，不推荐复杂工艺。

### 2. 风险信号

适合写成“看到什么，就要警惕什么”的预警。

示例：
- 上市时间未定但已进入打样，项目推进风险高。
- MOQ 未确认就做材质方案，报价误差大。

### 3. 流程更新

适合回写到 SOP、模板、审批点或开案清单的规则。

示例：
- 开案时必须确认实物规格、上市时间、MOQ 和环保优先级。
- 概念层后补一轮概念测试或货架模拟。

### 4. 品类洞察

适合跨项目复用的类别经验。

示例：
- 高端礼盒更依赖开箱节奏和触感，而不是堆叠工艺。
- 电商型包装优先看运输安全和装箱效率。

### 5. 待验证假设

值得保留，但证据不足以当成正式规则的结论。

要求：
- 标明低置信度
- 指出下一次应如何验证

## 共享知识库字段

每条共享知识至少包含：
- `ID`
- `类型`
- `结论`
- `适用场景`
- `建议动作`
- `证据来源`
- `置信度`

推荐展示结构：

- `决策规则`：`ID / 适用场景 / 判断规则 / 推荐动作 / 置信度 / 来源`
- `风险信号`：`ID / 预警信号 / 可能后果 / 建议动作 / 置信度 / 来源`
- `流程更新`：`ID / 应补什么 / 应修改位置 / 当前状态 / 来源`
- `品类洞察`：`ID / 适用场景 / 核心洞察 / 设计含义 / 置信度 / 来源`
- `待验证假设`：`ID / 假设 / 为什么值得保留 / 下一次怎么验证 / 当前证据`

## 回写规则

- 项目级总结先写 `knowledge-synthesis.md`
- 跨项目可复用内容再写入 `Workspace/Knowledge/`
- 共享知识库优先按分类文件维护，不额外制造同义文档
- 更新共享知识后，要同步更新 `Workspace/Knowledge/Library/knowledge-dashboard.md`、`Workspace/Knowledge/Library/workspace-knowledge-digest.md` 和 `Workspace/Knowledge/Library/knowledge-index.md`
- 如果某条知识本质是 skill 规则更新，标出应修改的 `SKILL.md`、`references/` 或 `assets/`
