# Structure Selector

## 概述

推荐能平衡保护性、展示感、装配复杂度和量产现实的结构方向。不要脱离上下文只给一个盒型名，要把取舍讲清楚。

## 典型触发句

- 这个产品适合什么盒型？
- 天地盖、抽屉盒和折叠盒，哪个更适合这次项目？
- 帮我看看这个包装需求应该怎么开合。
- 这个结构适合电商运输吗？
- 请帮我比较几个结构方案，并给出打样建议。
- 这个礼盒结构该怎么选？
- 先帮我比较几个盒型，看看哪个更稳。

## 使用流程

1. 找出当前最主导的约束：运输、陈列、礼赠、保护、装配还是成本。
2. 从 [structure-framework.md](./references/structure-framework.md) 里筛出 2 到 4 个合适结构。
3. 按产品重量、易碎度、尺寸、MOQ 和供应商能力进行比较。
4. 输出一个推荐方向、一个备选方向和打样计划。
5. 凡是依赖开模或供应商公差的假设都要明确标出。
6. 如果结构价值明显依赖开箱体验或供应商公差，按 [../packging-os/references/bridge-node-rules.md](../packging-os/references/bridge-node-rules.md) 判断是否应插入 `开箱测试` 或 `供应商可行性确认`。

## 评估规则

- 在讨论视觉戏剧性之前，先看保护性和装箱逻辑。
- 对于增加装配负担但用户价值不明显的结构，要降低优先级。
- 电商场景下，体积效率和防损是一级约束。
- 礼赠场景下，开箱节奏和价值感是一级约束。
- 如果产品偏重或易碎，在推荐硬盒或悬浮结构前，先确认承重和固定方式。
- 如果推荐理由主要来自开箱体验，而不是基础保护与装配逻辑，必须补桥接测试建议。

## 输出要求

- 输出比较表，写清优点、风险、成本影响和适用场景。
- 补充打样检查点和量产注意事项。
- 如果没有哪个结构明显安全可行，要直接说明，并写出还需要哪些尺寸或测试数据。

## 输出后检查

- 如果本次任务冻结了盒型、尺寸、公差假设、运输保护要求、结构风险或供应商/MOQ 前提，必须提示更新项目卡。
- 提示格式使用：`⚠️ 本次任务改变了 [具体项]，建议更新项目卡：project-memory-manager`

## 默认归档位置

- 默认目录：`03_Design/04_Production`
- 推荐文件名：`structure-decision.md`
- 归档规则：属于生产准备文档，可保留版本，不直接进入 `04_Final`

- 非标准结构（非天地盖/书型盒/抽屉盒等通用结构）必须触发供应商可行性确认桥接
- 结构方案必须包含装箱效率评估、打样周期预估、结构成本压力拆解
- 结构层不得直接判断单价，只能拆解成本压力来源：材料用量、灰板/卡纸/瓦楞等级、装配人工、刀模/开模、运输体积、损耗风险、MOQ 对单价的影响
- 输出表里的"成本影响"改为"成本压力与依据"，必须标注成本证据等级 [C0-C5]
