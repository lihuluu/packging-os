# Brief Decomposer

## 概述

把不完整的包装需求整理成**最低可推进 brief**。brief-decomposer 的工作不是把项目一开始就填成完美任务书，而是回答四件事：

1. 现在已知什么。
2. 当前到底要解决什么包装任务。
3. 哪些缺口会阻塞**下一步**。
4. 这些缺口应该由哪个后续 skill 接住。

brief 是入口文件，不是项目合同、结构方案、供应商 brief 或印前放行单。后续确认项应随项目推进，由对应 skill 写入自己的阶段文档，再同步回项目卡。

## Brief 成熟度

| 状态 | 定义 | 允许进入的内容 | 不应出现的内容 |
|------|------|----------------|----------------|
| 原始需求 | 只有一句话、聊天记录、销售转述或零散会议纪要 | 原话、明确事实、最小缺口 | 被包装成结论的人群、成本、结构、供应商、成功指标 |
| 可推进 | 已能支撑下一步动作，但仍有大量未知项 | 已知事实、当前任务、关键待确认、下一步移交 | 把后续阶段才会确认的事项写成开案必填 |
| 已确认 | 入口层信息已经被客户/内部确认，足以交给研究、概念、结构或供应商沟通 | 确认过的 brief 快照、冻结输入、移交边界、后续变更记录 | 内部推理、下游方案结论、未验证成本拆分、静默覆盖已冻结输入 |

## 典型触发句

- 帮我拆一下这个包装 brief。
- 客户只给了一段很模糊的话，你帮我整理成正式需求。
- 我有一份会议纪要，帮我提炼成包装项目 brief。
- 这个需求还缺什么关键信息，帮我列出来。
- 请把这段聊天记录转成包装设计任务书。
- 先帮我把这个需求捋顺。
- 这些零散笔记你帮我整理成能开工的 brief。

## 使用流程

1. 从用户输入或附件中提取已知事实。
2. 判断产品、当前任务和下一步要解锁的阶段。
3. 参考 [brief-framework.md](./references/brief-framework.md) 只追问缺失但影响最大的关键信息。
4. 生成轻量结构化 brief，并明确假设、风险和下一步决策点。
5. 把不属于 brief 阶段的事项放入“后续移交项”，不要在本阶段强行填满。
6. 结尾指出后续最应该进入的阶段：研究定位、概念、结构、视觉、供应商沟通或项目状态维护。

## 优先要问的问题

- 产品/SKU 是什么，属于什么品类？
- 当前要做的是新包装、改版、标签调整、发厂文件，还是先梳理方向？
- 这次为什么做：上新、改版、提价、渠道切换、礼赠、解决旧包装问题，还是内部提案？
- 这次包装最核心的诉求是什么？
- 已经固定的输入有哪些：品牌资产、规格、结构、文案、上市时间、渠道、供应商？
- 已知的调性关键词和视觉锚点分别是什么？
- 下一步要先做什么：研究、概念、结构、视觉、供应商可行性、印前合规，还是项目总卡？
- 哪 1-3 个缺口会阻塞下一步？

## 规则

- 在产品、任务和下一步目标没清楚前，不要直接跳进概念建议。
- 区分已知事实和推测。
- 先判断 brief 成熟度：原始需求 / 可推进 / 已确认。没有明确来源的信息，不得默认为已确认。
- 不要为了让 brief 好看而补满空项。真实早期 brief 可以有空白和待确认项。
- `packaging-brief.md` 是正式 brief 的唯一任务书模板输出；不要另起一个“正式 brief”文档。
- 原始需求和可推进阶段允许继续澄清、补充、修正；进入已确认状态后，`packaging-brief.md` 视为入口快照，应冻结当前版本。
- 已冻结 brief 后如果出现重大输入变化（产品/SKU、业务背景、核心诉求、渠道、规格、结构、时间、必须出现信息、设计范围），不要静默覆盖原内容；必须新增变更记录或版本说明，并标明变更来源。
- 成本拆分、MOQ、材料克重、供应商状态、验收负责人、素材授权负责人、合规负责人和成功指标默认是**后续移交项**；只有用户原文、附件、项目卡或后续确认明确给出时，才能写成 brief 已确认事实。
- 目标人群、竞品、工艺限制、排期倒推和成本压力可以作为工作假设，但必须放在 `brief-decomposition.md` 的假设、风险或移交项里，不直接写进 `packaging-brief.md` 当定论。
- 业务背景 / 为什么做、核心诉求、调性关键词、视觉锚点可以进入正式 brief，但必须区分确定性：用户明确表达的写入已确认；模型为了推进提出的只能放入假设或待确认边界。
- 调性关键词和视觉锚点必须拆开：调性关键词记录抽象感受词，视觉锚点记录可观察、可指向的参考物、材质、构图、色彩、竞品或文化线索。
- 尽早标出冲突，尤其是预算、时间线、结构野心和工艺野心之间的冲突。
- 如果涉及受监管品类，把合规信息标成后续 `prepress-checker` / 合规复核移交项；除非用户正在问发厂或上市，不要把合规负责人当作开案必填。
- 如果 brief 里没有渠道信息，可以继续形成原始需求或可推进 brief；只有当下一步要做结构、材质、供应商或印前判断时，才把渠道标为阻塞。
- 如果缺少合规负责人、素材授权负责人或样品验收负责人，只标为发厂/上市前阻塞项，并移交给 `project-memory-manager`、`visual-system-builder`、`supplier-brief-writer` 或 `prepress-checker`。

## 边界与移交

| 事项 | brief-decomposer 怎么处理 | 后续承接 skill |
| --- | --- | --- |
| 目标人群、竞品、定位 | 只记录已有信息或待研究问题 | `research-analyzer` |
| 概念方向、命名、故事 | 只记录客户原话和方向偏好 | `concept-generator` |
| 盒型、尺寸、运输保护 | 只记录已固定结构或尺寸缺口 | `structure-selector` |
| 正背面信息层级、视觉规则 | 只记录必须出现信息、调性关键词、视觉锚点和视觉避雷 | `visual-system-builder` |
| 纸张、工艺、成本压力 | 只记录预算目标或成本风险线索 [C0-C1] | `material-finishing-advisor` |
| MOQ、供应商、报价、打样口径 | 只记录是否已知，不生成正式询价 | `supplier-brief-writer` / `quotation-comparator` |
| 排期、责任人、审批链 | 只记录硬截止或已知责任人 | `project-tracker` / `project-memory-manager` |
| 合规、素材授权、样品验收 | 只记录已知风险和发厂/上市前阻塞 | `visual-system-builder` / `prepress-checker` |

## 输出要求

### 两份文档的分工

| 文档 | 定位 | 用途 |
|------|------|------|
| `brief-decomposition.md` | **过程分析文档** | 拆解、追问、风险、假设和移交项。是工作过程记录 |
| `packaging-brief.md` | **任务书快照** | 面向项目执行的正式文档。只同步已确认信息和明确待确认边界，不承载内部推理。状态可能是原始需求、可推进或已确认 |

### 同步规则

- brief-decomposer 负责**同时生成和维护**这两份文档，但只维护入口层事实、下一步移交边界和正式 brief 的版本/变更记录
- 用户每次确认入口层信息后，必须**同步更新 packaging-brief.md**，不能只更新 decomposition
- **packaging-brief.md 是任务书快照**，可能发给领导或客户，必须保持干净、严谨：
  - 文件顶部必须标注 Brief 状态：原始需求 / 可推进 / 已确认
  - 文件顶部必须标注版本、冻结状态、信息来源和最后更新
  - 不使用删除线标记已确认问题，直接更新为确认后的内容
  - 没有待确认的问题就去掉「待确认问题」章节
  - 不放内部流程建议（如「建议下一步」「推荐 skill」），这些只写在 decomposition 里
  - 使用表格呈现结构化信息，语气正式
- 如果 `packaging-brief.md` 已冻结，更新时必须保留或新增「变更记录」；不得直接把旧输入改没
- 不再要求预算、MOQ、上市时间、供应商状态、合规/标签负责人、素材授权负责人、样品验收负责人在开案 brief 一次性填满；这些只在影响下一步时标为阻塞，否则进入后续移交项
- 如果 brief 中缺少会阻塞下一步的关键项，输出中必须标注"下一步阻塞"；如果只影响发厂/上市，标注"发厂/上市前阻塞"
- 如果包装 brief 的状态落后于拆解表（比如拆解表已更新但 brief 还写着「未确认」），视为未完成

### 输出内容

**brief-decomposition.md（过程分析）**：
- Brief 状态、原始输入摘要、已知事实、当前任务、工作假设、关键缺口、后续移交项、风险、下一步问题
- 业务背景、核心诉求、调性关键词、视觉锚点如果信息不足，可以先作为待确认或工作假设，不要强行写成结论
- 问题要有优先级，控制在能解锁下一步判断的最小集合，默认不超过 3 个
- 明确推荐下一个适合衔接的 skill 或工作流

**packaging-brief.md（正式任务书）**：
- 使用 [packaging-brief-template.md](./assets/packaging-brief-template.md) 模板
- 只包含已确认的入口层事实、当前任务、业务背景、核心诉求、已知固定项、调性关键词、视觉锚点和明确待确认边界；未确认字段统一写「待确认」，不要用推测填空
- 如果 Brief 状态为已确认，标注冻结状态；后续变化写入变更记录
- 语气正式，适合直接发给领导或客户

## 输出后检查

- 如果本次 brief 拆解确认了项目名称、产品/SKU、规格、渠道、硬截止、固定结构、品牌资产或发厂/上市前阻塞项，必须提示更新项目卡。
- 提示格式使用：`⚠️ 本次任务改变了 [具体项]，建议更新项目卡：project-memory-manager`

## 默认归档位置

- 默认目录：`01_Brief`
- 推荐文件名：`brief-decomposition.md`（过程分析）+ `packaging-brief.md`（正式任务书）
- 归档规则：属于前期澄清文档，不进入 `04_Final`
- 两份文档始终成对存在于 `01_Brief/` 目录
