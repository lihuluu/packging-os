# Design Version Tracker

## 概述

把包装设计中的"决策"变成可追溯的记录。它不是文件版本控制系统（不复制 PSD/AI 文件），而是**决策日志工具**。

核心原则：
- **决策日志是答案**。真正要解决的是"为什么改了"。
- **追加-only,永不覆盖**（仅指 `## 决策记录` 表行）。决策记录表只追加新行，历史行不动；同文件内的其他字段块允许更新。
- **5 列精简格式**：编号、变更内容、触发来源、决策理由、日期。
- **零迁移负担**。现有项目完全不触碰，新功能通过 `init-project.py` 和模板自然扩散。

## 典型触发句

- 帮我记一下这次变更的理由。
- 这个设计为什么改成这样？
- 帮我追溯这个决策是怎么来的。
- 这个工艺为什么从烫金改成UV？
- 记录一下客户这次的反馈和对应的修改。

## 使用流程

1. **识别请求类型**：
   - 记录决策理由 → `decision-log.md` 追加条目
   - 追溯某个决策 → 在 `decision-log.md` 中按编号或关键词检索

2. **填写决策日志**：
   - 每次重要设计决策后（客户反馈驱动、打样问题驱动、成本压力驱动等），在 `decision-log.md` 追加一行
   - 必填：编号、变更内容、触发来源、决策理由、日期
   - 编号使用 `D-001`、`D-002` 格式，同项目内全局递增
   - 如果缺少必填信息，先追问；用户无法补全时，只把待确认信息写入 `开放问题` 或 `关键假设`，不要伪造进决策表

3. **追溯决策**：
   - 按触发来源（客户反馈/打样问题/成本压力/法规变更）筛选
   - 按时间线回溯
   - 必要时关联到 `project-memory-card.md` 的 `本次更新` 字段交叉验证

## 决策日志触发时机

**必须记录**（每次发生都应追加）：
- 客户反馈导致的设计变更
- 打样/彩样发现问题后的修正
- 成本超出预算后的工艺/结构调整
- 法规/合规要求变化导致的修改
- 供应商能力限制导致的工艺替换
- 桥接节点结论（概念测试、货架模拟、开箱测试、概念到材质翻译、供应商可行性确认等）

**建议记录**：
- 内部评审中的重大取舍（如两个结构方案的最终选择理由）
- 视觉方向上的关键转折（如从 A 概念切换到 B 概念的原因）
- 材质/工艺组合的最终锁定理由

**无需记录**：
- 微调类修改（字号调整 2pt、色值微调）
- 排版对齐修正
- 纯装饰性元素增减

## 输出要求

- 决策日志：`## 决策记录` 表行追加 only，不覆盖历史行；其余字段块允许更新
- 每次输出必须区分 `已知事实` / `假设` / `建议`
- 涉及成本、工艺或量产变化的决策，必须附风险说明
- 不确定信息不得写成已确认决策；需要保留时写入 `关键假设` 或 `开放问题`

## 默认归档位置

- 决策日志：`03_Design/01_Working/decision-log.md`
