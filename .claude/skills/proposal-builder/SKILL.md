# Proposal Builder

## 概述

把多个阶段已经形成的判断整理成一次可评审、可汇报、可推进决策的提案结构。重点不是重新发明内容，而是把已有结论组织成顺序、论点和推荐路径。

## 典型触发句

- 帮我整理这个包装提案逻辑。
- 把这些研究、概念和结构判断串成一个提案页序。
- 这套包装方案怎么讲给客户或老板更顺？
- 帮我写这次包装提案的讲解备注。
- 请把这些方向整理成可评审的提案大纲。

## 使用流程

1. 收集已经确认的 brief、研究结论、概念方向、结构判断、视觉规则和材质建议。
2. 参考 [proposal-framework.md](./references/proposal-framework.md) 判断本次提案是“方向选择型”还是“方案确认型”。
3. 按问题、洞察、方案、理由、风险、待拍板事项组织页序。
4. 输出提案大纲；需要时再补讲解备注和可能被问到的问题。
5. 结尾写清本次提案要拍板什么，以及提案后该进入哪个下一个 skill。

## 组织规则

- 先讲为什么这样做，再讲做成什么样。
- 先收束推荐路径，再展示备选，不把听众丢进无序对比。
- 不把还没验证的假设包装成既定事实。
- 如果推荐依赖供应商、成本或打样验证，要把这些依赖单独列出来。
- 如果当前资料不足以支撑提案结论，要先指出缺口，而不是硬拼一个完整故事。

## 输出要求

- 输出 `提案目标`、`页序大纲`、`每页核心信息`、`推荐路径`、`可能被问到的问题` 和 `待拍板事项`。
- 需要讲稿时，再补 `讲解备注`。
- 结尾补充 `已知事实`、`关键假设`、`主要风险` 和 `建议衔接的下一个 skill`。

## 输出后检查

- 如果本次提案整理冻结了推荐路径、改变了阶段结论、暴露了新的风险/阻塞、或确认了预算、时间线、供应商与法规前提，必须提示更新项目卡。
- 提示格式使用：`⚠️ 本次任务改变了 [具体项]，建议更新项目卡：project-memory-manager`

## 默认归档位置

- 默认目录：`03_Design/03_Presentation`
- 推荐文件名：`proposal-outline.md`
- 归档规则：属于提案与评审文档，可保留版本，不直接进入 `04_Final`

- 支持阶段性汇报（不只做最终提案）：概念汇报、结构+材质汇报、设计终稿汇报
- 阶段性汇报必须包含当前商业风险和待确认项
