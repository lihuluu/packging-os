## 阶段识别

Packging OS 的 9 层工作流保持通用，当前默认业务域是茶叶包装设计。用户未说明品类时，按茶叶 SKU、茶礼盒、茶叶标签、茶叶包装供应链和茶叶包装版本项目判断；用户明确提出非茶叶品类时，先保留通用流程，再标注缺少该品类专用规则。

在提额外问题前，先根据当前产物判断阶段：
- brief 文本、会议纪要、模糊目标：洞察层
- 竞品图、截图、市场笔记：洞察层或概念层
- 命名草案、故事方向、创意关键词：概念层
- 刀模、盒型草图、尺寸限制、运输说明：结构层
- 视觉参考、风格板、版式草图、字体/图片/插画授权状态：视觉层
- 材料样、印刷规范、工艺需求、法规说明：材料层
- 印前 PDF、校样、色差问题、样品缺陷、合规复核、验收记录、成本目标：印前层
- 提案需求、供应商邮件、报价对比、项目排期：商业交付层
- 销售反馈、复盘笔记、用户投诉、上市回顾：复盘层
- 跨项目经验整理、共享知识库维护、规则回写：复盘层

## 分层地图

### 0. 状态层
- `SK-00`：维护项目卡
- `SK-00B`：记录关键决策
- `SK-00C`：决策追溯

适用场景：
- 新项目开案
- 多轮评审后需要更新统一状态
- 跨 skill 切换时需要避免上下文丢失
- 记录设计变更的决策理由和触发源

产出：
- 项目总卡
- 决策记录
- 决策日志 (`decision-log.md`)

### 1. 洞察层
- `SK-01`：拆解 brief
- `SK-02`：分析竞品
- `SK-03`：理解消费者与使用场景
- `SK-04`：定义定位与价值主张

适用场景：
- 需求还很模糊
- 还没进入形式设计，需要先把方向定清楚

产出：
- 项目需求拆解表
- 包装定位摘要
- 机会点地图

### 2. 概念层
- `SK-05`：生成创意概念
- `SK-06`：命名与文案
- `SK-07`：可视化概念方向

适用场景：
- 需要包装方向、命名、概念故事或情绪表达

产出：
- 概念方向卡
- 命名与卖点文案选项
- 给后续视觉使用的情绪线索
- 概念验证计划（可选）

### 3. 结构层
- `SK-08`：选择盒型或主结构
- `SK-09`：定义刀模与结构限制
- `SK-10`：规划打样与验证
- `SK-11`：探索结构创新

适用场景：
- 讨论盒型、开合方式、保护性、装配方式或运输逻辑

产出：
- 结构决策表
- 打样检查单
- 结构风险说明

### 4. 视觉层
- `SK-12`：把品牌语言转译成商业沟通目标
- `SK-13`：安排版式与信息层级
- `SK-14`：选择色彩与对比逻辑
- `SK-15`：定义视觉规则
- `SK-16`：明确效果图与商业场景验证需求
- `SK-16B`：登记字体、图片、插画、AI 图和品牌资产授权状态
- `SK-16C`：拆解 AI 概念素材任务并归档生成草案

适用场景：
- 结构已知，接下来要定义包装各面在真实购买场景里承担什么任务，以及看起来是什么样
- 视觉方案使用了字体、图片、插画、AI 生成图、认证标志或合作方 Logo
- 视觉方向单中的图像/图形需要由 AI 生成提案草案、方向验证图或设计师重绘参考

产出：
- 视觉方向单
- 正背侧面任务分配与信息层级说明
- 效果图与场景验证需求单
- 素材授权登记 (`asset-register.md`)
- AI 概念素材任务和项目内素材草案 (`03_Design/02_Assets/AI_Concepts/{task-id}/`)

### 5. 材料层
- `SK-17`：选择材质
- `SK-18`：选择印刷方式
- `SK-19`：选择后道工艺
- `SK-20`：检查环保与法规要求

适用场景：
- 需要决定纸材、板材、塑料替代、印刷方式或工艺组合

产出：
- 材质与工艺建议单
- 可行性与成本说明
- 合规风险提醒

### 6. 印前层
- `SK-21`：准备量产文件
- `SK-22`：规划校样与色彩控制
- `SK-23`：执行质检与量产检查
- `SK-24`：在不破坏设计意图下做成本优化
- `SK-24B`：检查合规复核、白样/彩样/大货验收状态

适用场景：
- 设计即将进入打样或量产
- 用户在问校样、修正、缺陷或降本问题
- 用户在问白样、彩样、大货首件、验收或商业销售前是否可放行

产出：
- 印前检查单
- 打样问题诊断
- 合规检查记录 (`compliance-review.md` 或并入 `prepress-review.md`)
- 样品与大货验收记录 (`sample-acceptance-record.md`)
- 成本优化建议

### 7. 商业交付层
- `SK-25`：写供应商 brief
- `SK-26`：比较供应商报价
- `SK-27`：准备提案
- `SK-28`：管理时间线与依赖关系

适用场景：
- 正式供应商 brief、报价对比、项目排期、提案汇报和审批交付
- 注意：商业检查不是第 7 层才开始的，每层都有对应的横向商业检查点（见下方）

产出：
- 供应商沟通单
- 报价对比表
- 提案大纲
- 讲解备注
- 行动清单

### 8. 复盘层
- `SK-29`：追踪上市反馈
- `SK-30`：沉淀经验并更新系统
- `SK-31`：把项目经验回写到共享知识库

适用场景：
- 产品已经上市，或项目周期已经结束

产出：
- 项目复盘记录
- 问题归因图
- 面向未来项目的 SOP 更新项
- 项目知识沉淀摘要
- 共享知识库更新项

## 各层打法

### 状态层打法

要问：
- 当前阶段是什么？
- 哪些信息已经确认？
- 哪些是假设或待确认项？
- 这次更新后，哪些内容发生了变化？

重点：
- 用统一项目卡承接所有阶段产出
- 让后续 skill 不需要重新拼背景

### 洞察层打法

要问：
- 产品/SKU 是什么，当前包装任务是什么？
- 哪些输入已经固定，哪些只是客户偏好或销售转述？
- 下一步要先进入研究、概念、结构、视觉、供应商沟通还是印前处理？
- 哪 1-3 个缺口会阻塞下一步？

重点：
- 把模糊需求转成最低可推进 brief
- 不把 MOQ、供应商、材质、合规负责人、验收责任人和成功指标强行塞进前期 brief
- 对下游才会确认的事项，写成移交项，而不是入口层结论
- 若进入研究输出，必须标注证据等级和样本分类，不允许出现无证据标注的结论

### 概念层打法

要问：
- 情绪基调更偏高端、活泼、专业、礼赠、自然还是技术感？
- 名称是开放的还是已固定？
- 哪些卖点或信息必须上包装？

重点：
- 生成 3 到 5 个真正不同的方向，而不是微调版
- 每个概念都要和人群、渠道、购买逻辑挂钩
- 新品牌、抽象表达或高风险方向，概念层后优先插入 `概念测试`、`货架模拟` 或供应商可行性确认，再进入下游阶段
- 如果概念已经明显依赖特殊触感、复杂工艺或结构体验，概念层结束时同步输出 `概念到材质翻译` 或 `概念到结构影响`

### 结构层打法

要问：
- 主导使用场景是运输、陈列还是礼赠？
- 产品重量、脆弱度和装配容错如何？
- 是否有尺寸或装箱限制？

重点：
- 平衡保护性、体验感和生产复杂度
- 凡是依赖供应商开模或工艺能力的假设都要标出来

### 视觉层打法

要问：
- 哪些面是关键决策面：正面、顶部、侧面还是内页？
- 哪些信息必须出现？
- 哪些品牌资产是强制的，哪些可选？
- 字体、图片、插画、AI 生成图、认证标志和合作方 Logo 是否有商用授权或内部资产依据？

重点：
- 先建层级，再做装饰
- 视觉规则必须和结构、印刷方式相容
- 任何权利不明的素材只能进入参考或草案，不能进入最终交付

### 材料层打法

要问：
- 单个包装的成本上限是多少？
- 供应商真实支持哪些材质和工艺？
- 是否涉及食品接触、标签法规、回收要求或地区规范？茶叶项目重点看内袋食品接触、防潮防串味、茶罐/铝箔袋/礼盒组合和标签信息区。

重点：
- 推荐可落地的材质与工艺组合
- 明确手感、成本、速度、环保之间的取舍
- 优先承接概念层给出的关键词，把抽象方向翻译成具体材质特征和验证项

### 印前层打法

要问：
- 量产使用什么文件格式和色彩流程？
- 当前能做哪种校样：数码样、合同样还是机上样？
- 目前已经出现了哪些问题或变更？
- 法规/标签信息是否已由内部法规、品控或负责人复核？
- 白样、彩样、机上样、大货首件或抽检现在处于什么验收状态？

重点：
- 降低可预防的生产事故
- 在减少返工和浪费的同时保住设计意图
- 未完成合规复核、素材授权和样品验收的项目，不应标记为可进入最终交付或量产闭环

### 商业交付层打法

要问：
- 谁负责审批什么？
- 哪个供应商负责哪个环节？
- 当前哪个里程碑最有风险？
- 当前是需要询价、打样、提案还是推进时间线？

重点：
- 把设计判断转成可以沟通和执行的文档
- 把待定事项和已确认事项分开
- 先确保沟通口径一致，再推进供应商或内部审批
- 如果一次命中多个协作任务，先按 `询价/打样口径 -> 报价比较 -> 项目推进 -> 提案整理` 排顺序，不要求用户自己拆技能

### 复盘层打法

要问：
- 市场、生产和内部协作分别发生了什么？
- 哪些问题是系统性，哪些只是单项目事故？
- 下一次开案时，哪些问题必须提前加入清单？

重点：
- 把观察转成可复用规则
- 用能改进未来路由和判断的形式沉淀结论
- 对跨项目可复用的结论，继续写入 `Workspace/Knowledge/Library/`
- 批量整理时，先做覆盖盘点，再做知识抽取

## 快速路由

常见请求可直接映射：
- “帮我整理一下这个包装项目当前状态” -> 状态层
- “帮我拆这个包装 brief” -> 洞察层
- “给我 3 个包装方向” -> 概念层
- “这个适合抽屉盒还是天地盖？” -> 结构层
- “正面应该怎么做层级？” -> 视觉层
- “该用什么纸和工艺？” -> 材料层
- “帮我写一份给供应商的打样 brief” -> 商业交付层
- “比较一下这几家工厂的报价” -> 商业交付层
- “帮我排一下项目推进时间线” -> 商业交付层
- “帮我检查这个文件能不能印” -> 印前层
- “总结一下这次上市的经验” -> 复盘层
- “把这个项目的工作产出沉淀成知识库” -> 复盘层
- “这个概念落到什么材质会更对？” -> 材料层
- “这个方向要不要先做概念测试？” -> 概念层

## 交接规则

只有当前层产出明确后，才进入下一层：
- 状态层 -> 任意层：项目卡已足够支持下一步判断
- 洞察层 -> 概念层：定位语句已经稳定 + 概念层已执行证据接收检查（见 concept-generator 接收规则）
- 概念层 -> 结构层：已选出一个主方向，且高风险方向已完成概念测试、货架模拟或并行可行性确认
- 概念层 -> 材料层：如果概念高度依赖触感、纸张或工艺信号，先输出 `概念到材质翻译`
- 结构层 -> 视觉层：关键面和核心约束已固定
- 视觉层 -> 材料层：视觉意图已足够清楚，能进入可行性测试
- 材料层 -> 印前层：材质与印刷假设已经确认
- 材料层 -> 商业交付层：已足够支持询价、打样或时间线推进
- 印前层 -> 商业交付层：量产信息已经能被他人正确理解
- 复盘层 -> 洞察层：把新的问题和规则写回开案清单

## 横向商业检查点

供应商沟通、成本意识和排期风险不是第 7 层才开始的。每一层都有自己的商业检查项：

| 层级 | 商业检查点 | 成本依据等级 |
|------|-----------|-------------|
| 0. 状态层 | MOQ、供应商状态、审批人、硬截止是否已记录 | 预算上限标注 [C1]？ |
| 1. 洞察层 | 渠道、预算、批量、上市时间、供应商是否已定 | 预算/成本上限的证据等级？ |
| 2. 概念层 | 用户共鸣、货架/电商表现、供应链依赖是否验证 | 成本变量已识别 [C0]？ |
| 3. 结构层 | 装箱效率、打样周期、供应商能力、结构成本 | 成本压力来源已拆解 [C0-C2]？ |
| 4. 视觉层 | 电商缩略图、货架远视、印刷适配、法规信息区、素材授权 | 印刷复杂度成本压力 [C0]？ |
| 5. 材料层 | MOQ、纸材稳定性、工艺良率、成本/交期、食品接触材料状态 | 供应商报价 [C3+] 或成本压力 [C0]？ |
| 6. 印前层 | 校样审批、色差、文件责任边界、合规复核、样品验收 | 最终成本已确认 [C4-C5]？ |
| 8. 复盘层 | 哪些商业风险应前置进下一次开案 | 成本估算与实际偏差？ |

**关键原则**：当用户问"现在要不要先找供应商确认？"时，系统不能因为还没到第 7 层而推迟协作判断。如果当前方案影响成本、MOQ、交期或供应商能力，可以在任意层插入 `供应商可行性确认` 桥接。

## 桥接节点

桥接节点的完整触发条件、默认顺序和跳过规则，统一以 [bridge-node-rules.md](./bridge-node-rules.md) 为准。

这里仅保留工作流层约束：

- 概念层后的高风险切换，先看是否需要 `概念测试`、`货架模拟` 或 `概念到材质翻译`
- 结构与礼赠体验相关时，先看是否需要 `开箱测试`
- 进入供应商或量产决策前，先看是否需要 `供应商可行性确认`
