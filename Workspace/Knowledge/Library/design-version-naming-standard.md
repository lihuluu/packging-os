# Design Version Naming Standard

本规范用于茶叶包装设计文件的轻量版本管理，目标是让设计师快速知道：当前是什么版本、处于什么阶段、能不能用于打样或印刷，以及历史版本为什么被替代。

## Known Facts

| 项目 | 规则 |
| --- | --- |
| 命名参考 | `HT7156 藏金纳福（手筑茯茶）950克_V1.1_FINAL_20251027` |
| 当前采用 | 产品信息 + 版本号 + 阶段 + 日期 |
| 未转曲区分方式 | 靠文件夹路径和文件格式区分，不靠文件名状态后缀区分 |
| 印刷文件格式 | 发厂印刷文件必须为 PDF |
| 适用范围 | 茶叶包装设计稿、评审稿、打样稿、印刷稿和历史归档稿 |

## Naming Rule

```text
{产品编号} {产品名称（品类/工艺）}{规格}_V{主版本}.{次版本}_{阶段}_{YYYYMMDD}
```

示例：

```text
HT7156 藏金纳福（手筑茯茶）950克_V1.1_PRINT_20251027.pdf
```

## Stage Definitions

| 阶段 | 文件标识 | 用途 | 是否可生产 |
| --- | --- | --- | --- |
| 草稿 | `DRAFT` | 设计师内部探索版本 | 否 |
| 内审 | `REVIEW` | 给市场、产品、法务、老板看的版本 | 否 |
| 打样 | `PROOF` | 已确认准备打样的版本 | 仅可打样 |
| 印刷 | `PRINT` | 最终用于印刷厂生产的版本 | 是 |
| 归档 | `ARCHIVE` | 已被替代或停用的历史版本 | 否 |

## Version Rules

版本号和阶段必须由设计师手工确认。Packaging OS 不自动判断当前文件处于哪个版本，也不替代设计师保存、命名、移动或发布设计文件。

| 变化类型 | 是否升版本 | 示例 |
| --- | --- | --- |
| 内部微调、排版对齐、轻微调色 | 不升版本 | 继续用当前工作文件 |
| 文案、卖点、信息层级、主视觉、工艺小调整 | 次版本 +0.1 | `V1.1` -> `V1.2` |
| 政策法规、产品定位、包装结构、年度改版等大调整 | 主版本 +1 | `V1.2` -> `V2.0` |
| 只改变阶段，不改变设计内容 | 不升版本，只改阶段 | `V1.2_REVIEW` -> `V1.2_PROOF` |

## Daily Workflow

| 场景 | 推荐动作 | 示例 |
| --- | --- | --- |
| 设计师内部探索 | 只维护工作文件 | `HT7156 藏金纳福（手筑茯茶）950克_WORKING.ai` |
| 发内部评审 | 导出 `REVIEW` | `HT7156 藏金纳福（手筑茯茶）950克_V1.1_REVIEW_20251018.pdf` |
| 确认打样 | 保存 `PROOF` | `HT7156 藏金纳福（手筑茯茶）950克_V1.1_PROOF_20251022.ai` |
| 最终印刷 | 输出 `PRINT` PDF | `HT7156 藏金纳福（手筑茯茶）950克_V1.1_PRINT_20251027.pdf` |
| 旧版停用 | 移入归档或标记 `ARCHIVE` | `HT7156 藏金纳福（手筑茯茶）950克_V1.0_ARCHIVE_20251027.ai` |

## Directory Rules

| 文件类型 | 默认路径 | 说明 |
| --- | --- | --- |
| 日常工作文件 | `Workspace/Projects/{项目名称}/03_Design/01_Working/` | 设计师内部滚动修改 |
| 评审展示文件 | `Workspace/Projects/{项目名称}/03_Design/03_Presentation/` | PDF/JPG，用于内部或客户评审 |
| 打样准备文件 | `Workspace/Projects/{项目名称}/03_Design/04_Production/` | 准备给供应商打样 |
| 发厂生产文件 | `Workspace/Projects/{项目名称}/04_Final/01_Print_Files/` | 已确认的印刷 PDF 文件 |
| 未转曲源文件 | `Workspace/Projects/{项目名称}/04_Final/02_Source_Files/` | 可编辑留底，不直接发厂 |
| 预览确认文件 | `Workspace/Projects/{项目名称}/04_Final/04_Previews/` | PDF/JPG 预览确认 |
| 旧版归档 | `Workspace/Projects/{项目名称}/03_Design/01_Working/Archive/` | 被替代或停用的历史版本 |

## Source File Handling

未转曲文件不加额外后缀，仍然使用同一套命名，靠目录和文件格式区分。发厂印刷文件必须输出为 PDF，未转曲可编辑源文件保留为 AI 等源文件格式。

| 类型 | 路径 | 文件名示例 |
| --- | --- | --- |
| 未转曲可编辑源文件 | `04_Final/02_Source_Files/` | `HT7156 藏金纳福（手筑茯茶）950克_V1.1_PRINT_20251027.ai` |
| 发厂印刷文件 | `04_Final/01_Print_Files/` | `HT7156 藏金纳福（手筑茯茶）950克_V1.1_PRINT_20251027.pdf` |

## Responsibility Boundary

| 工作 | 责任人 | Packaging OS 的作用 |
| --- | --- | --- |
| 保存设计源文件、导出 PDF、移动到对应目录 | 设计师 | 提供目录与命名规则 |
| 判断当前阶段是 `DRAFT / REVIEW / PROOF / PRINT / ARCHIVE` | 设计师 | 按设计师确认的事实写入记录 |
| 判断当前版本号是 `V1.2` 还是 `V1.3` | 设计师 | 校验命名是否符合规则 |
| 判断文件是否已经法务、老板、市场或供应商确认 | 设计师 | 记录确认状态和变更理由 |
| 记录关键变更原因 | 设计师提供事实，Packaging OS 整理 | 写入 `decision-log.md` 和项目总卡 |
| 发厂前检查 PDF | 设计师提供文件，Packaging OS 辅助 | 使用 `prepress-checker` 做印前检查 |

## Decision Logging

以下变化必须进入 `decision-log.md`：

| 必须记录 | 不必记录 |
| --- | --- |
| 客户/老板/市场/法务反馈导致修改 | 字号微调 |
| 政策法规导致修改 | 对齐、间距微调 |
| 打样发现问题后修改 | 普通装饰元素轻微变化 |
| 成本压力导致工艺或材质调整 | 内部探索废稿 |
| 供应商能力限制导致替换方案 | 未对外发送的尝试稿 |

Packaging OS 推荐记录位置：

| 文档 | 默认路径 | 作用 |
| --- | --- | --- |
| `decision-log.md` | `03_Design/01_Working/` | 记录关键设计决策和变更原因 |

## Assumptions

| 假设 | 说明 |
| --- | --- |
| 目录语义能被团队遵守 | 未转曲与发厂文件不靠后缀区分，必须依赖固定目录和文件格式 |
| `PRINT` 等同于可用于生产的阶段 | 如果历史文件仍使用 `FINAL`，可在过渡期视为 `PRINT` |
| 工作文件可以滚动修改 | 内部探索不要求每次导出正式版本 |

## Risks

| 风险 | 说明 | 控制方式 |
| --- | --- | --- |
| 源文件误发工厂 | 未转曲 AI 被当成印刷文件发送 | 发厂压缩包只从 `04_Final/01_Print_Files/` 打包，且只包含 PDF |
| PDF 输出错误 | 印刷 PDF 可能缺出血、字体、专色或工艺标注 | 发厂前用 `prepress-checker` 做印前检查 |
| 版本号滥用 | 小改也升版导致版本爆炸 | 只有影响判断、合规、生产、成本的变化才升版 |
| 只改文件不记原因 | 后期不知道为什么改 | 关键变更进入 `decision-log.md` |
| 让系统猜版本 | Packaging OS 无法知道设计软件里当前打开或发出的真实文件 | 版本号、阶段和确认状态必须由设计师手工确认 |
